# AGENTS.md — Kino

Guidance for AI agents working on this codebase.

## What This Project Is

Kino is a local-first personal finance desktop app. Users track accounts, transactions, budgets, and net worth. All data lives in a `.kino` SQLite file on the user's machine. There is no backend server, no sync service, no accounts.

Tech stack: **Go + Wails v2** (desktop shell) + **Vue 3 + TypeScript + Vite + Tailwind** (frontend) + **SQLite** (storage).

## Architecture Rules

- **Cents everywhere.** All monetary values are `int64` cents. Never use `float64` for money.
- **Single writer.** SQLite is opened with `SetMaxOpenConns(1)` and WAL mode. Do not add concurrent writers.
- **No remote calls.** This app is intentionally offline-first. Do not add HTTP calls to external services.
- **Wails RPC boundary.** Public methods on the `App` struct in `app*.go` files are automatically exposed as JavaScript. Return types must be JSON-serializable. Errors are returned as the second return value (`error`).
- **Frontend bindings are generated.** Files under `frontend/src/wailsjs/` are auto-generated by `wails dev` / `wails build`. Do not edit them manually.
- **Schema migrations are idempotent.** All DDL changes go through the migration system in `internal/db/`. Each migration checks the current `schema_version` in `kino_meta` before applying. Increment the version.
- **Transfer pairs.** A transfer between two accounts creates two linked transaction rows joined by `TransferPairID`. Both rows must be created/deleted/updated together.
- **Foreign keys are on.** SQLite is opened with `PRAGMA foreign_keys = ON`. Respect referential integrity.

## Key Data Types (internal/models/)

- `Account` — id, name, type, institution, balance_cents, currency, is_hidden, sort_order
- `Transaction` — id, account_id, date, amount_cents, payee, category_id, is_transfer, transfer_pair_id, is_reconciled, import_hash
- `Category` — id, name, parent_id, color, icon, is_income, is_system, sort_order
- `Budget` — id, category_id, amount_cents, period (monthly/weekly/annual), rolls_over, start_date
- `Goal` — id, name, type, target_cents, current_cents, linked_account_id, target_date

## Common Patterns

**Adding a new App method:**
1. Add the method to the appropriate `app_*.go` file (or create a new one).
2. The method must be on the `App` struct and exported.
3. Run `wails dev` to regenerate frontend bindings in `frontend/src/wailsjs/`.
4. Call from Vue via the generated binding, e.g. `import { MyMethod } from '../wailsjs/go/main/App'`.

**Adding a DB column / table:**
1. Add a new migration in `internal/db/` that checks `schema_version` before running.
2. Increment the version constant.
3. Update the relevant `models/` struct.
4. Update any queries that SELECT or INSERT the affected table.

**Modifying import logic:**
- CSV parser: `internal/importer/csv.go`
- OFX parser: `internal/importer/ofx.go`
- Duplicate detection (exact hash + fuzzy): `app_import.go`

## What to Avoid

- Do not use `float64` for any monetary computation.
- Do not edit `frontend/src/wailsjs/` — it is generated.
- Do not open additional SQLite connections; reuse `app.db`.
- Do not add external API calls or telemetry.
- Do not break the transfer-pair invariant (both sides must stay in sync).
